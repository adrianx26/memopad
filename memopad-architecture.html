MemoPad Architecture

Local-First Knowledge Management System

What is MemoPad?

MemoPad is a local-first knowledge management system that lets you build persistent knowledge through natural conversations with LLMs like Claude. Everything stays in Markdown files on your computer while the system maintains a searchable knowledge graph.

1. System Overview

MemoPad creates a bridge between human editors and AI assistants. Both read and write Markdown files, while a sync engine maintains a searchable database index.

2. Data Flow (File Operations)

The sync engine keeps files and database synchronized bidirectionally. 

    sequenceDiagram
      actor Human
      participant File as "coffee.md"
      participant Watch as "File Watcher"
      participant Sync as "Sync Service"
      participant Parser as "Markdown Parser"
      participant DB as "SQLite"

      Note over Human,DB: Human edits file
      Human->>File: Edit content
      File-->>Watch: File changed event
      Watch->>Sync: Trigger sync

      Sync->>File: Read markdown
      Sync->>Parser: Parse content

      Note right of Parser: Extract:
- Frontmatter
- Observations
- Relations

      Parser-->>Sync: Entity object
      Sync->>DB: Upsert entity
      Sync->>DB: Update observations
      Sync->>DB: Update relations
      Sync->>DB: Update FTS index

      Note over Human,DB: AI reads via MCP
      actor AI
      AI->>Sync: read_note("coffee")
      Sync->>DB: Query entity
      DB-->>Sync: Entity + Obs + Rels
      Sync-->>AI: Formatted response

      Note over Human,DB: AI writes via MCP
      AI->>Sync: write_note(title, content)
      Sync->>DB: Create entity
      Sync->>File: Write markdown
      File-->>Human: Visible in editor
Key Point: Files are the source of truth. The database is just an index for search and fast lookups. Any change to files triggers a sync; any AI write updates both DB and files.

3. Knowledge Graph Structure

Three core concepts form the knowledge graph. Each Markdown file becomes an entity with observations (facts) and relations (links). 

    erDiagram
      ENTITY {
        string title "Coffee Brewing"
        string permalink "coffee-brewing"
        string file_path "coffee.md"
        datetime created_at
        datetime updated_at
      }

      OBSERVATION {
        string category "method"
        string content "Pour over extracts more..."
        string tags "brewing"
      }

      RELATION {
        string relation_type "relates_to"
        string target "bean-origins"
        string context "see also"
      }

      ENTITY ||--o{ OBSERVATION : "has many"
      ENTITY ||--o{ RELATION : "links via"
      RELATION }o--o| ENTITY : "connects to"
# Example: coffee.md --- title: Coffee Brewing permalink: coffee-brewing tags: [coffee, brewing] --- ## Observations (facts) - [method] Pour over extracts more floral notes - [technique] Water temp 205°F is optimal #brewing - [preference] Ethiopian beans have bright flavors ## Relations (links) - relates_to [[Bean Origins]] - requires [[Grinding Technique]] - pairs_well_with [[Chocolate]]

Key Design Principles

Local-First

Markdown files are the source of truth. You own your data. The database is just an index for search and performance.

Bi-Directional

Both humans and AI assistants read and write to the same files. No separate "AI knowledge base" — just shared Markdown.

Composition Root Pattern

Each entrypoint (API, MCP, CLI) has its own container that wires dependencies explicitly. No hidden global state.

Semantic Markup

Simple Markdown patterns unlock rich knowledge graphs. Observations with [category] and Relations with [[WikiLinks]].

Three Entrypoints

API for REST clients, MCP for AI assistants, CLI for terminal users. All share the same service layer.

Explicit Dependencies

Configuration flows through composition roots. Services receive dependencies explicitly, not through globals.
